# v2026.5.12 Release Announcement

**Release Date:** 2026-05-25

Fortémi v2026.5.12 ships the first realtime provider milestone: Twilio Voice call ingestion, Deepgram live speech-to-text, provider-neutral realtime contracts, and a durable batch transcription bridge for completed call recordings.

## Highlights

| Feature | Description |
|---------|-------------|
| **Twilio Voice Realtime Path** | Signed Voice webhooks create call sessions and Twilio Media Streams audio reaches Fortemi over `/api/v1/realtime/twilio/{CallSid}`. |
| **Provider-Neutral Contracts** | Call lifecycle, media frames, ASR events, and outbox rows use stable Fortemi contracts while Twilio wire details stay inside the adapter. |
| **Deepgram Streaming ASR** | Live audio streams can produce partial/final transcript events with reconnect, fallback accounting, and health metrics. |
| **Recording Batch Transcription** | Twilio recording callbacks import audio into file storage and queue the existing `AudioTranscriptionHandler` for higher-quality post-call transcript output. |
| **Fail-Closed Authentication** | API authentication is now on by default unless an operator explicitly acknowledges anonymous local mode. |

---

## Realtime Provider Integration

This release establishes the realtime provider architecture described in the RTP ADRs. The implementation intentionally keeps provider-specific wire formats at the adapter boundary:

- Twilio Voice webhook fields are translated into Fortemi call lifecycle events.
- Twilio Media Streams envelopes are normalized into `MediaFrame` values before ASR.
- Transcript and call events are written through the provider-neutral `event_outbox` contract.
- Future providers can implement the same adapter contracts without changing downstream transcript consumers.

The first production adapter is Twilio Voice; the mock adapter remains in-tree for deterministic CI coverage and contract parity tests.

## Live And Batch Transcripts

The release supports the hybrid transcription pattern:

1. During the call, Deepgram streaming ASR produces live partial/final transcript events.
2. When Twilio later sends `recording.completed`, Fortemi imports the recording as an audio attachment.
3. Fortemi queues the existing `AudioTranscriptionHandler` using attachment IDs, so the batch path reuses the mature audio transcription pipeline.
4. Call-session metadata records the batch policy as `append_attachment_transcript`, linking live call data and durable attachment transcript output.

This keeps live latency and final transcript quality separate without creating a second transcription subsystem.

## Security And Consent

Authentication now defaults to fail-closed for `/api/v1/*`. Anonymous local mode requires both:

```bash
REQUIRE_AUTH=false
I_UNDERSTAND_NO_AUTH=true
```

Multi-tenant deployments refuse anonymous mode regardless of acknowledgment.

For Twilio, webhook signatures are verified with the receiver secret and the externally visible URL. Reverse proxies must preserve `X-Forwarded-Host` and `X-Forwarded-Proto` when Fortemi is behind TLS termination.

Operators remain responsible for call recording and live transcription consent. The realtime setup guide documents disclosure, explicit confirmation, audit metadata, and regulated deployment considerations.

## Upgrade Notes

1. **Run migrations normally.** This release adds incoming webhook receiver, realtime call-session, and event-outbox tables.
2. **Review auth configuration.** Deployments relying on the previous anonymous default must explicitly set the local-mode acknowledgment pair.
3. **Configure file storage before using recording batch transcription.** The `recording.completed` bridge needs file storage and network access to the Twilio recording URL.
4. **Configure public HTTPS/WSS reachability for Twilio.** Twilio must reach both the Voice webhook receiver and the media WebSocket endpoint.
