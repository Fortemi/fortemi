## Stateless BYOK Inference Endpoints (#628)

Three new endpoints enable downstream UIs and integrations to route chat completions through FortĂ©mi as a CORS-bypassing proxy. Credentials are passed per-request in the body and never stored server-side.

### `POST /api/v1/inference/complete`

Provider-agnostic chat completion. Accepts an optional override block:

```json
{
  "prompt": "Summarize this note",
  "provider_id": "openai",
  "api_key": "<OPENAI_API_KEY>",
  "base_url": "https://api.openai.com/v1"
}
```

Falls back to registered config and environment variables when the override block is absent. Supports `ollama`, `openai`, `openrouter`, and `llamacpp`. `ProviderRegistry::resolve_generation_inline()` builds a fresh `Box<dyn GenerationBackend>` per call, never mutating the shared registry.

### `POST /api/v1/inference/stream`

SSE equivalent of `/complete`. Returns a stream of `delta` events followed by a `done` event. Ollama emits one event per token (see below); other backends use a single-chunk fallback until streaming overrides are added.

### `GET /api/v1/inference/providers`

Lists registered providers with their availability flags:

```json
[
  { "id": "ollama",      "server_configured": true,  "requires_user_key": false },
  { "id": "openai",      "server_configured": false, "requires_user_key": true  },
  { "id": "openrouter",  "server_configured": false, "requires_user_key": true  }
]
```

BYOK UIs use this to determine which provider keys to prompt for.

## Real Token Streaming for Ollama (#629)

`/api/v1/inference/stream` now emits one SSE `delta` event per token for Ollama backends instead of buffering the full response.

`GenerationBackend` trait gains two streaming methods:

```
async fn stream_generate(&self, prompt: &str) -> Result<GenerationStream>
async fn stream_generate_with_system(&self, system: &str, prompt: &str) -> Result<GenerationStream>
```

where `GenerationStream = Pin<Box<dyn Stream<Item = Result<String>> + Send>>`.

`OllamaBackend` overrides both with a real implementation that POSTs to `/api/chat` with `stream=true` and yields each `message.content` chunk from the NDJSON response as a separate stream item. All other backends get a default one-element wrapper around the existing blocking `generate()` â€” wire-compatible without code changes.

**Practical effect:** HotM and other Matric platform clients connected to Ollama will see progressive token output. OpenAI/OpenRouter streaming overrides are follow-up work; the fallback keeps them functional in the interim.

## Periodic Inference Provider Reprobe (#630)

Previously, `capabilities.chat.available` and `capabilities.inference.available` latched to `false` when a provider was unreachable at startup, requiring a FortĂ©mi restart to clear.

This release replaces the one-shot startup probe with a background task:

- Re-checks reachability every `INFERENCE_PROBE_INTERVAL_SECS` (default: `30`).
- `AppState.inference_available: Arc<AtomicBool>` tracks current state and is read by the health endpoint and chat handler.
- Chat returns `503 Service Unavailable` with a `retry_after` field while the provider is unreachable.
- `InferenceAvailabilityChanged` SSE event fires on every `true â†’ false` and `false â†’ true` transition â€” subscribed UIs toggle their offline banners without polling.

## Caller-Defined Extraction Pipeline

Notes can now opt in to specific AI processing stages via a `pipeline` field on `CreateNoteRequest`:

```json
{
  "title": "My note",
  "content": "...",
  "pipeline": ["revision", "title_generation", "concept_tagging"]
}
```

Supported stages: `revision`, `title_generation`, `concept_tagging`, `reference_extraction`, `metadata_extraction`, `document_type_inference`.

An empty array stores the note with zero AI processing. Omitting `pipeline` runs the full default pipeline â€” fully backwards compatible.

Two new environment variables replace hardcoded constants in the archive extraction adapter:

| `ARCHIVE_MAX_EXTRACT_BYTES` | 1 GB | 100 MB |
| `ARCHIVE_MAX_SINGLE_FILE_BYTES` | 50 MB | 10 MB |

The `MAX_FILES` per-archive cap has been removed entirely.

## Fixes

### Revision mode "none" no longer creates fake history (#625)

Notes created or updated with `revision_mode=none` previously wrote a `note_revision` record with the rationale "Original preserved (no AI revision)" â€” polluting revision history with meaningless entries. The new `sync_revised_to_original_tx` keeps FTS content in sync without inserting any revision record.

### Redis connection timeout (#624)

`ConnectionManager::new()` now times out after 5 seconds when Redis is unreachable, preventing server startup from hanging indefinitely. The bundle image defaults to `REDIS_ENABLED=false` so Redis unavailability at startup is never a startup blocker.

### arm64 optional components (#623)

New Docker build args â€” `ENABLE_OPEN3D`, `ENABLE_POSTGIS`, `ENABLE_OCR`, `ENABLE_FFMPEG` â€” allow disabling platform-specific components that fail to compile or run on arm64. Enables clean cross-architecture builds without patching the Dockerfile.

### Think tokens no longer block streaming

Disabled Qwen3.5 `think` mode during streaming requests so content `delta` events flow immediately. Previously the stream would stall until the entire `<think>â€¦</think>` block completed.

### `:latest` Docker tag integrity

Dev pushes to `main` no longer overwrite `:latest` and `:bundle-latest` on GHCR. From this release:

- `:latest` / `:bundle-latest` â€” tagged releases only (set by the release CI job on `v*` tags)
- `:main` / `:bundle-main` â€” latest commit on `main` (rolling dev builds)
- `:sha-<short>` â€” immutable per-commit reference

**Recommended pinning:**
- Rolling dev: `ghcr.io/fortemi/fortemi:bundle-main`
- Pinned release: `ghcr.io/fortemi/fortemi:bundle-v2026.4.2`
