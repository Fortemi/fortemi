# v2026.4.2 Release Announcement

**Release Date:** 2026-04-22

Fortémi v2026.4.2 delivers **real-time inference streaming**, **bring-your-own-key (BYOK) inference endpoints**, and **auto-recovering provider health** — focused entirely on making the inference layer more capable and more resilient for downstream UIs and integrations.

## Highlights

| Feature | Description |
|---------|-------------|
| **BYOK Inference Endpoints** | `/complete`, `/stream`, and `/providers` let any client drive chat completions through Fortémi without storing keys server-side |
| **Real Token Streaming** | Ollama backends now emit one SSE `delta` per token — per-token visibility in HotM, G0DM0D3, and Crucible UIs |
| **Periodic Provider Reprobe** | Inference providers that come back online after startup are detected automatically within 30s |
| **SSE Availability Events** | `InferenceAvailabilityChanged` events let UIs toggle offline banners without polling `/health` |
| **`:latest` Tag Integrity** | CI change reserves `:latest` on GHCR exclusively for tagged releases, ending masked-release incidents |

## Stateless BYOK Inference Endpoints (#628)

Three new endpoints enable downstream UIs and integrations to route chat completions through Fortémi as a CORS-bypassing proxy. Credentials are passed per-request in the body and never stored server-side.

### `POST /api/v1/inference/complete`

Provider-agnostic chat completion. Accepts an optional override block alongside the prompt:

```json
{
  "prompt": "Summarize this note",
  "provider_id": "openai",
  "api_key": "sk-...",
  "base_url": "https://api.openai.com/v1"
}
```

Falls back to the registered provider config and environment variables when the override block is absent or partial. Supports `ollama`, `openai`, `openrouter`, and `llamacpp`.

`ProviderRegistry::resolve_generation_inline()` builds a fresh `Box<dyn GenerationBackend>` from transient credentials on each call, never mutating the shared registry between requests.

### `POST /api/v1/inference/stream`

SSE equivalent of `/complete`. Returns a stream of `delta` events as content is generated, followed by a `done` event. Ollama backends emit one event per token (see below). Other backends use a single-chunk fallback until streaming overrides are added.

### `GET /api/v1/inference/providers`

Lists all registered inference providers with their availability flags:

```json
[
  { "id": "ollama", "server_configured": true, "requires_user_key": false },
  { "id": "openai", "server_configured": false, "requires_user_key": true },
  { "id": "openrouter", "server_configured": false, "requires_user_key": true }
]
```

BYOK UIs use this response to determine which provider keys to prompt for from the user.

## Real Token Streaming for Ollama (#629)

`/api/v1/inference/stream` now emits one SSE `delta` event per token for Ollama backends instead of buffering the full response into a single chunk.

The `GenerationBackend` trait gains two streaming methods:

```
async fn stream_generate(&self, prompt: &str) -> Result<GenerationStream>
async fn stream_generate_with_system(&self, system: &str, prompt: &str) -> Result<GenerationStream>
```

where `GenerationStream = Pin<Box<dyn Stream<Item = Result<String>> + Send>>`.

`OllamaBackend` overrides both with a real implementation that POSTs to `/api/chat` with `stream=true` and yields each `message.content` chunk from the NDJSON response as a separate stream item. All other backends get a default implementation that wraps the existing blocking `generate()` in a one-element stream — wire-compatible without code changes.

**Practical effect:** HotM, G0DM0D3, and Crucible UIs connected to Ollama providers will see progressive token output. OpenAI/OpenRouter streaming overrides are left as follow-up work; the fallback keeps them functional in the interim.

## Periodic Inference Provider Reprobe (#630)

Previously, `capabilities.chat.available` and `capabilities.inference.configured` latched to `false` when a provider was unreachable at startup — requiring a Fortémi restart after the provider came back online.

This release replaces the one-shot startup probe with a background task:

- A spawned probe re-checks provider reachability every `INFERENCE_PROBE_INTERVAL_SECS` (default: `30`).
- `AppState.inference_available: Arc<AtomicBool>` tracks current reachability and is updated on each probe cycle.
- `/health` and `/health/live` expose `capabilities.inference.available` reflecting the live probe result.
- The chat handler returns `503 Service Unavailable` with a `retry_after` field when the provider is temporarily unreachable rather than failing with a misleading error.
- An `InferenceAvailabilityChanged` SSE event fires on every `true → false` and `false → true` transition, letting subscribed UIs toggle their "provider offline" banners without polling `/health`.

## Fixes

### Think tokens no longer block streaming

Qwen3.5's `think` mode is now disabled during streaming requests. Previously, a streaming response would stall until the entire `<think>…</think>` block completed before any content delta was emitted — defeating the purpose of streaming for thinking-capable models.

### `:latest` Docker tag integrity

The CI pipeline was overwriting `:latest` and `:bundle-latest` on GHCR with every push to `main`, causing downstream consumers pinned to `:latest` to silently receive dev builds. This masked v2026.4.1 after it was published.

From this release forward:
- `:latest` and `:bundle-latest` are written **only** by the release publish job, triggered by a version tag (`v*`).
- Dev pushes write `:main` and `:sha-<short>` exclusively.

**Recommended pinning:**
- Rolling dev builds: `ghcr.io/fortemi/fortemi:bundle-main`
- Pinned releases: `ghcr.io/fortemi/fortemi:bundle-v2026.4.2` (once tagged)

## Upgrade Notes

1. **No database migrations** — this release adds no new tables or schema changes.
2. **`INFERENCE_PROBE_INTERVAL_SECS`** — new optional env var (default: `30`). Increase to reduce probe overhead on stable setups; decrease for faster recovery detection.
3. **`:latest` pin change** — if you were relying on `:latest` for rolling dev builds, switch to `:bundle-main`.
4. **Streaming clients** — the `/stream` endpoint now sends multiple `delta` events for Ollama. Clients that assumed a single `delta` per request must handle the multi-event pattern. The `done` event shape is unchanged.
