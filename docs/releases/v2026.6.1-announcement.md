# v2026.6.1 Release Announcement

**Release Date:** 2026-06-30

Fortémi v2026.6.1 is a hardening and contract-cleanup release. It standardizes the API error surface on **RFC 9457 `problem+json`**, completes a broad **secret/diagnostic redaction** pass across errors and logs, adds **HTTP security headers**, and moves the **incoming-webhook receiver to HMAC-only** authentication. Documentation is reconciled to the shipped code.

## Highlights

| Change | Why You Care |
|--------|--------------|
| **RFC 9457 error contract** *(breaking)* | Every API error now returns `application/problem+json` with a stable `type` URI, `title`, `status`, `detail`, and a request id — machine-parseable, consistent, and published in the OpenAPI spec. Legacy `{"error", "message", "details"}` bodies are gone. |
| **Redaction hardening** | Error responses and logs no longer leak filenames, paths, secrets, or internal diagnostics across the API, core, jobs, and crypto surfaces. |
| **Security headers** | All HTTP responses carry standard security headers. |
| **Webhook HMAC auth** | The incoming-webhook receive endpoint authenticates via HMAC signature — no bearer token required. |
| **Scope separation** | The MCP scope is separated from REST read/write scopes; realtime transport (SSE/WebSocket) requires the MCP scope. |
| **Fail-closed startup** | The API refuses to start on invalid configuration rather than running in a degraded state. |

## Breaking change: RFC 9457 error contract (#967)

All API error paths — auth, OAuth, PKE, chat/SSE streaming, archive routing, ingest streaming, and inference — now return `application/problem+json`:

```json
{
  "type": "https://fortemi.com/problems/validation-error",
  "title": "Validation Error",
  "status": 422,
  "detail": "…",
  "request_id": "…"
}
```

The RFC 6749 OAuth error code is used internally to select the problem `type`; it is no longer emitted as a top-level `error` field. Clients that parsed the old `{"error": …}` shape must switch to reading `type`/`status`/`detail`. See `docs/content/api-error-contract.md` for the full problem-type catalog.

Rate limiting is a single global limiter; the 429 response is `problem+json` (`type=.../rate-limit-exceeded`) with no `X-RateLimit-*` headers — clients back off on 429.

## Security hardening (#968, #974)

A broad redaction pass ensures error responses, `Debug`/`Display` output, and logs never expose secret-shaped values, tokens, DSNs, filesystem paths, or internal diagnostics. The `anyhow` dependency is bumped to 1.0.103 to clear RUSTSEC-2026-0190.

## Documentation

The `docs/content/` set was reconciled to the shipped code: RFC 9457 error responses, the real inference endpoints (`/complete`, `/stream`, `/config`, `/providers`, `/test-connection`), MCP tool counts (43 core / 205 full) and annotation reality, and corrected environment variables (`MATRIC_INFERENCE_DEFAULT`, default generation model `qwen3.5:9b`).

## Upgrade notes

- **Update error handling** in any client that parsed the legacy `{"error", "message", "details"}` shape — switch to RFC 9457 `problem+json` (`type`/`status`/`detail`).
- **Incoming webhooks**: remove any bearer token on the receive endpoint; sign requests with the configured HMAC secret instead.
- **Realtime clients** (SSE/WebSocket): ensure the token carries the MCP scope.
