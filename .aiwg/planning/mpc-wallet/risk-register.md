## Scoring Legend

| Rating | Impact | Likelihood |
|--------|--------|------------|
| **Critical** | Complete identity compromise, irrecoverable data loss, or regulatory violation | Near-certain (>80%) |
| **High** | Partial key compromise, extended downtime, or significant user impact | Likely (50-80%) |
| **Medium** | Degraded functionality, workaround available, limited user impact | Possible (20-50%) |
| **Low** | Minor inconvenience, no security impact, cosmetic | Unlikely (<20%) |

**Exposure** = Impact x Likelihood (Critical > High > Medium > Low)

## Risk Register

| Risk ID | Description | Impact | Likelihood | Exposure | Mitigation Strategy | Owner | Status | Residual Risk |
|---------|-------------|--------|------------|----------|---------------------|-------|--------|---------------|
| R-001 | **MPC protocol implementation bug leaks key material.** A flaw in the FROST DKG or signing implementation causes a participant to reveal their secret share or the combined private key to be reconstructable from protocol messages. | Critical | Medium | Critical | (1) Use the `frost-secp256k1` crate from the ZCash Foundation FROST suite, which has formal specification alignment. (2) Never implement custom MPC primitives; wrap audited library functions only. (3) Add property-based tests (proptest) asserting that no combination of protocol messages from fewer than t participants can recover the group secret. (4) Engage third-party cryptographic audit before production release. | Crypto Lead | Open | Medium -- residual risk from crate bugs upstream, mitigated by audit and fuzz testing. |
| R-002 | **Same-key-for-sign-and-encrypt cross-protocol attack.** Reusing the same key material for both FROST threshold signing (secp256k1) and X25519 ECDH encryption enables cross-protocol oracle attacks where signing queries leak information usable to break encryption, or vice versa. | Critical | Low | High | (1) Enforce strict key separation: MPC wallet uses secp256k1 exclusively for signing; existing X25519 keys handle encryption. No shared key material between curves or protocols. (2) Use distinct key derivation paths: signing keys derive from `m/44'/60'/0'` (Roko-compatible), encryption keys derive from `m/73'/0'/0'` (custom). (3) Code review checklist item: verify no function accepts both signing and encryption key types. | Crypto Lead | Open | Low -- key separation eliminates the attack vector entirely if enforced. |
| R-003 | **Side-channel attack on threshold signing.** Timing or power analysis on the device performing FROST partial signing reveals the secret share, particularly on mobile devices without constant-time guarantees. | High | Low | Medium | (1) Use constant-time implementations from `frost-core` which delegates to `k256` (RustCrypto) with documented constant-time scalar operations. (2) Add timing variation tests that run signing with different share values and assert bounded execution time variance. (3) For high-security deployments, recommend hardware security module (HSM) or secure enclave for share storage. (4) Document that mobile platforms have weaker side-channel resistance as a known limitation. | Crypto Lead | Open | Low -- constant-time library code on server/desktop is well-defended; mobile remains a known residual. |
| R-004 | **Weak entropy during DKG ceremony.** Insufficient randomness during distributed key generation produces predictable polynomial coefficients, allowing an attacker who observes the ceremony to reconstruct shares. | Critical | Low | High | (1) Use `OsRng` (operating system CSPRNG) for all DKG randomness, never userspace PRNGs. (2) Add entropy quality assertion at DKG start: read 32 bytes from `OsRng`, verify non-zero and non-repeating. (3) On Linux, block if `/dev/urandom` is not yet seeded (check `getrandom` flag). (4) Log entropy source metadata in ceremony audit record (OS, RNG backend). (5) Reject DKG initiation on platforms where entropy quality cannot be verified. | Crypto Lead | Open | Low -- `OsRng` on supported platforms (Linux, macOS, Windows) provides cryptographic-grade entropy. |
| R-005 | **FROST crate has unaudited code path.** The `frost-secp256k1` crate (ZCash Foundation) has had partial audit but specific code paths (e.g., identifiable abort, DKG round 2 verification) may not be covered. Undiscovered bugs in these paths could compromise ceremony integrity. | High | Medium | High | (1) Pin to a specific audited release tag; do not track `main`. (2) Monitor ZCash Foundation security advisories and crate changelogs. (3) Wrap all FROST calls in an abstraction layer (`matric-crypto::mpc` module) so upstream can be swapped without API changes. (4) Contribute fuzz targets upstream for unaudited paths. (5) Budget for independent audit of our integration layer and the specific FROST code paths we invoke. | Crypto Lead | Open | Medium -- audit coverage will improve over time but never reach 100% for a moving target. |
| R-006 | **User loses 2+ devices (below threshold), identity unrecoverable.** In a 2-of-3 scheme, losing two devices simultaneously (theft, fire, hardware failure) drops below the signing threshold. The user's MPC wallet identity is permanently lost with no recovery path. | Critical | Medium | Critical | (1) Require one of the three shares to be a "recovery share" stored as an encrypted backup (Argon2id + AES-256-GCM, minimum 128-bit passphrase entropy). (2) Recovery share stored in user-controlled location (e.g., password manager, safety deposit box) separate from devices. (3) Provide guided setup flow that refuses to finalize DKG until recovery share is exported and user confirms storage. (4) Support optional social recovery: user can designate 3-of-5 trusted contacts who each hold a Shamir share of the recovery share. (5) Document that losing both devices AND the recovery share is unrecoverable by design (no backdoor). | Product Lead | Open | Medium -- relies on user discipline for recovery share storage; social recovery adds a safety net. |
| R-007 | **MPC signing latency exceeds Roko 100ms tick window over WAN.** FROST two-round signing requires message exchange between threshold participants. Over WAN with typical 50-150ms RTT, the total signing ceremony may exceed the Roko block production window, causing transaction inclusion delays. | Medium | High | Medium | (1) Pre-compute FROST nonce commitments (Round 1) during idle periods and cache them. With pre-computed nonces, signing requires only Round 2 (single message exchange). (2) Target architecture: device keys sign independently using delegated device certificates for routine operations; MPC ceremony reserved for commissioning actions (key rotation, trust attestation, device delegation). (3) Benchmark signing latency across realistic network conditions; set SLA at p99 < 500ms for commissioning operations (which are not time-critical). (4) Implement timeout with retry for signing rounds; stale nonces are discarded and regenerated. | Platform Lead | Open | Low -- commissioning operations are infrequent and tolerant of higher latency. |
| R-008 | **Trust graph fragmentation -- islands of disconnected users.** If users only attest trust to close contacts, the trust graph develops disconnected components. Users in small clusters cannot verify identities outside their cluster, reducing the network's utility. | Medium | High | Medium | (1) Design trust attestation UX that encourages broader connections (e.g., "verify 3 people you know" onboarding flow). (2) Implement graph connectivity monitoring: alert when connected component count exceeds threshold relative to total users. (3) Support introduction attestations: if A trusts B and B trusts C, B can introduce A to C (but A must independently verify and attest -- no automatic transitive trust). (4) Provide public directory of self-attested identities as discovery mechanism (opt-in). (5) Accept that small deployments will have sparse graphs; design all features to work without trust graph (graceful degradation). | Product Lead | Open | Medium -- fragmentation is inherent in decentralized trust; mitigations reduce but cannot eliminate it. |
| R-009 | **First-contact key exchange over insecure channel (MITM).** When two users establish initial trust, if they exchange public keys over an unauthenticated channel (email, messaging), an attacker can substitute their own keys and intercept all subsequent communication. | Critical | Medium | Critical | (1) Require out-of-band verification for first-contact: display short auth string (SAS) derived from both parties' public keys; users confirm match over voice/video/in-person. (2) Implement QR code key exchange for in-person verification. (3) Anchor first-contact public keys to Roko temporal receipts -- users can verify key age and detect recent substitution. (4) Display visual key fingerprint (SSH-style randomart or emoji hash) prominently in UI. (5) Log and surface key change events: if a contact's key changes, require re-verification with prominent warning. | Crypto Lead | Open | Medium -- SAS verification is effective but depends on user completing the step; UI must make it frictionless. |
| R-010 | **Device cert expiry during network partition.** A device certificate expires while the device is offline or unable to reach the issuing MPC wallet (e.g., network outage, airplane mode). The device loses signing capability until reconnection and re-delegation. | Medium | Medium | Medium | (1) Default cert lifetime of 24 hours with configurable extension up to 72 hours for travel scenarios. (2) Device proactively renews cert when 50% of lifetime has elapsed (12 hours before expiry by default). (3) Implement grace period: expired certs accepted for up to 1 hour past expiry with reduced privilege (read-only operations). (4) Cache last-known-good cert chain locally so device can present proof of prior authorization even after expiry. (5) Queue operations performed during partition for replay once connectivity restored and cert renewed. | Platform Lead | Open | Low -- proactive renewal and grace period handle most real-world scenarios. |
| R-011 | **Multi-curve maintenance burden (4 curves, 2+ MPC protocols).** Supporting secp256k1 (Roko signing), X25519 (encryption), Ed25519 (potential future), and P-256 (external interop) across FROST and existing ECDH creates a large cryptographic surface area with compounding audit and maintenance costs. | High | High | High | (1) Strict scope: launch with exactly two curves -- secp256k1 (MPC signing via FROST) and X25519 (encryption, existing). No Ed25519 or P-256 until concrete requirement emerges. (2) Abstract curve operations behind a `CurveOps` trait so implementations are isolated. (3) Each curve module has independent test suite with known-answer tests from RFC/specification test vectors. (4) Document the curve decision in an ADR with criteria for adding new curves (must have: concrete use case, audited Rust implementation, test vectors). (5) Annual review of curve necessity; remove unused curves. | Crypto Lead | Open | Medium -- two curves is manageable; risk re-emerges if scope creeps to additional curves. |
| R-012 | **Backward compatibility break with existing PKE system.** Introducing MPC wallet concepts may disrupt the existing MMPKE01 format, X25519 key storage, or encryption/decryption workflows for current users. | High | Medium | High | (1) MPC wallet is additive: new `MMPKE02` format version supports MPC-derived recipient keys alongside existing X25519 keys. (2) Existing `MMPKE01` files remain readable and writable without MPC wallet. (3) Migration path: users opt in to MPC wallet; their existing X25519 encryption key is preserved as a standalone key. (4) Feature flag `mpc_wallet` gates all new code paths; disabled by default in initial releases. (5) Integration tests verify that all existing PKE test fixtures pass unchanged after MPC code is merged. | Platform Lead | Open | Low -- additive design with feature flag eliminates forced migration. |
| R-013 | **Database migration complexity for new trust/MPC tables.** The MPC wallet requires new tables (device_certs, trust_attestations, mpc_shares, signing_ceremonies) and potentially new enum types. Complex migrations risk conflicting with the existing 57+ migration files and multi-schema architecture. | Medium | Medium | Medium | (1) Group all MPC tables into a single migration file with carefully chosen timestamp (verify no duplicates via `ls migrations/ \| sed 's/_.*//' \| sort \| uniq -d`). (2) No `CREATE INDEX CONCURRENTLY` (sqlx transaction constraint). (3) No `ALTER TYPE ... ADD VALUE` in migration transactions; use separate migration file if new enum values needed. (4) MPC tables follow existing per-memory schema pattern (cloned into archive schemas via `clone_schema`). (5) Test migration on a copy of production database before release. | Platform Lead | Open | Low -- following established migration patterns reduces risk significantly. |
| R-014 | **Roko RPC dependency creates single point of failure for temporal anchoring.** If the Roko blockchain node is unreachable, all operations requiring temporal receipts (trust attestation anchoring, key rotation proofs) are blocked. | High | Medium | High | (1) Temporal anchoring is asynchronous: operations proceed immediately, receipt is fetched and attached in background job. (2) Implement receipt cache: store recent temporal receipts locally; validate against Roko on next available connection. (3) Support multiple Roko RPC endpoints with failover (primary + 2 fallbacks). (4) Define maximum acceptable receipt age (e.g., 1 hour); operations without fresh receipt are flagged but not blocked. (5) All Roko-dependent features degrade gracefully: trust attestations are valid without temporal anchor but display "unanchored" status in UI. | Platform Lead | Open | Medium -- graceful degradation prevents blocking but reduces security guarantees when Roko is unavailable. |
| R-015 | **Self-signed device certs rejected by external systems.** Device certificates issued by the MPC wallet are self-signed (no CA hierarchy). External systems or partner integrations may reject them, limiting interoperability. | Medium | Medium | Medium | (1) Device certs are scoped to Fortemi ecosystem operations; external systems use standard OAuth2/API key auth (existing system). (2) Document that device certs are for internal device delegation only, not external identity proof. (3) If external interop is required, bridge via Roko temporal receipt: the receipt is verifiable by any party with Roko chain access. (4) Provide cert export in standard X.509 format for systems that accept custom CAs, with the MPC wallet's group public key as the issuer. | Product Lead | Open | Low -- clear scope boundary prevents misuse; Roko bridge provides external verifiability. |
| R-016 | **Key recovery mechanism creates backdoor concern.** The encrypted recovery share (R-006 mitigation) could be perceived as or actually become a backdoor if the encryption is weak, the passphrase is guessable, or the storage location is compromised. | High | Low | Medium | (1) Recovery share encrypted with Argon2id (memory=256MB, iterations=4, parallelism=4) + AES-256-GCM. (2) Enforce minimum passphrase entropy at creation: reject passphrases below 128 bits estimated entropy (zxcvbn score 4). (3) Recovery share file contains no metadata identifying the user or wallet -- only the encrypted share blob. (4) Document threat model: recovery share compromise allows 1-of-3 shares; attacker still needs a second share. (5) Optional: split recovery share itself with 2-of-3 Shamir for paranoid users. | Crypto Lead | Open | Low -- high-entropy passphrase + Argon2id makes brute force infeasible; 1-of-3 is insufficient alone. |
| R-017 | **No established legal framework for self-sovereign identity.** Self-sovereign identity and MPC-based key management exist in a regulatory gray area. Future regulations could mandate key escrow, government-accessible recovery, or prohibit certain cryptographic operations. | Medium | Low | Low | (1) Design with regulatory optionality: the MPC architecture does not preclude adding authorized recovery parties if legally mandated. (2) Monitor regulatory developments (EU eIDAS 2.0, US digital identity proposals). (3) Document compliance posture: Fortemi is a personal knowledge base, not a financial or identity provider; MPC wallet is for data signing, not financial transactions. (4) Legal review before marketing any identity-related claims. | Product Lead | Open | Low -- personal knowledge base use case is low regulatory risk; architecture allows adaptation. |
| R-018 | **FROST crate dependency unmaintained or compromised.** The `frost-secp256k1` crate could become unmaintained (ZCash Foundation shifts priorities) or suffer a supply chain compromise (malicious release). | High | Low | Medium | (1) Pin exact crate version with `=x.y.z` in Cargo.toml; never use range specifiers for crypto dependencies. (2) Verify crate checksum against known-good value in CI (`cargo-audit` + `cargo-vet`). (3) Monitor crate publish events via crates.io RSS/API. (4) Maintain fork capability: `matric-crypto::mpc` abstraction layer allows swapping to a fork or alternative FROST implementation. (5) If crate is unmaintained for >6 months with known issues, evaluate `givre` or `multi-party-ecdsa` as alternatives. (6) Vendor the crate source in `vendor/` as insurance. | Crypto Lead | Open | Medium -- supply chain risk is inherent in the Rust ecosystem; mitigations reduce but cannot eliminate it. |
| R-019 | **Breaking changes in Roko temporal receipt format.** Roko network upgrades could change the temporal receipt structure (fields, encoding, signature scheme), breaking Fortemi's receipt validation and storage. | Medium | Medium | Medium | (1) Abstract receipt handling behind `TemporalReceipt` trait with versioned deserialization (support receipt_v1, receipt_v2, etc.). (2) Store raw receipt bytes alongside parsed fields so re-parsing is possible after format updates. (3) Subscribe to Roko governance proposals that affect receipt format; target 2-sprint lead time for format migrations. (4) Include Roko chain spec version in stored receipts for future disambiguation. (5) Receipt validation includes format version check; unknown versions trigger warning, not hard failure. | Platform Lead | Open | Low -- versioned abstraction handles format evolution cleanly. |

## Appendix: Assumptions

1. FROST threshold scheme is 2-of-3 (t=2, n=3) for user devices.
2. Roko Network uses ECDSA secp256k1 with PoAT consensus.
3. Existing PKE system (MMPKE01, X25519/AES-256-GCM) continues to operate independently.
4. Device certificates are short-lived (24h default) and issued by the MPC wallet's group key.
5. Trust attestations are P2P (no central authority) and anchored to Roko temporal receipts.
6. The `frost-secp256k1` crate from the ZCash Foundation FROST suite is the primary MPC dependency.
