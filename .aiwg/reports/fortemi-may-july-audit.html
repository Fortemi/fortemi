CRITICAL Schema & Migrations CONFIRMED migrations/20260215000000_native_uuidv7.sql:20

Already-applied migration 20260215000000_native_uuidv7.sql was edited in place — sqlx checksum validation will refuse to start every existing deployment on upgrade

Evidence

Commit 10d2601f (2026-05-17, 'feat(db): declare PostgreSQL 18 as hard requirement (#635)') added 25 lines to a migration originally created in commit 42f88e98 (Feb 2026, timestamp prefix 20260215): a new 'DO $pg18_guard$ BEGIN IF current_setting(''server_version_num'')::int < 180000 THEN RAISE EXCEPTION ...' block at lines 20-27 plus rewritten header comments. crates/matric-db/src/lib.rs:315-321 runs 'sqlx::migrate!("../../migrations").run(&self.pool)' with default checksum validation; grep across crates/ and docker/bundle-entrypoint.sh finds no checksum-repair path (no UPDATE of _sqlx_migrations.checksum anywhere). Any deployment that applied the original file between Feb and May 2026 (including release-pinned bundle-latest hosts) will fail startup with sqlx MigrateError::VersionMismatch ('migration 20260215000000 was previously applied but has been modified'). This is the exact known-pitfall documented in the project's own conventions, and the edit is redundant anyway — Database::migrate() already calls require_postgres_18() (lib.rs:316) before migrations run.

Impact

On the next image upgrade, every pre-May deployment's API container crash-loops at boot: sqlx compares the stored checksum in _sqlx_migrations against the modified file and aborts the whole migrator before any pending migration runs. New installs are unaffected, making this invisible in CI (fresh DB) — it only detonates in the field.

Fix

Revert 20260215000000_native_uuidv7.sql to its byte-exact pre-10d2601f content and move the PG18 guard into a NEW migration with a fresh timestamp (or rely solely on require_postgres_18() in Database::migrate, which already covers the startup path). If reverting is no longer possible because some deployments applied the modified version, ship a one-time entrypoint repair that updates _sqlx_migrations.checksum for version 20260215000000 to match the canonical file before sqlx::migrate! runs. Add a CI check that fails if a committed migration file's content changes after it first appears on main.

Standard sqlx::migrate::Migrator docs — applied migrations are checksum-validated and 'previously applied but has been modified' is a hard error (MigrateError::VersionMismatch); Flyway/sqlx migration best practice: never edit an applied migration, always add a new one.
CRITICAL Realtime & Telephony CONFIRMED crates/matric-api/src/main.rs:4131

Unauthenticated /api/v1/ws WebSocket streams the full cross-tenant event bus (note titles, tags, filenames) even when REQUIRE_AUTH=true

Evidence

Route `/api/v1/ws` is registered at main.rs:3440 and classified `RealtimeTransport` in route_policy.rs:1422-1427. `is_public_without_bearer` treats that class as fully public: route_policy.rs:1500-1512 `matches!(policy.class, Public | Docs | OAuth | RealtimeTransport | SystemHealth)`. So `auth_middleware` (main.rs:6810 `if is_auth_exempt(&method,&path){return next.run(request).await;}`) bypasses auth entirely, and `authorize_middleware` then early-returns because no `Auth` extension was injected. The handler itself performs ZERO auth: `async fn ws_handler(ws, State(state))` (main.rs:4127) → `handle_ws_connection` (main.rs:4131) immediately `let mut event_rx = state.event_bus.subscribe();` (4137) and forwards `envelope.payload` for every event (4149-4151) with no token, scope, or memory filter — unlike the sibling SSE handler which does authenticate. An anonymous client connecting to wss://host/api/v1/ws receives a live firehose of every tenant's note titles, tags, attachment filenames, collection/archive names and job activity, defeating multi-tenant isolation and the fail-closed auth posture.

Remove `RealtimeTransport` from the public set in `is_public_without_bearer`, or add in-handler bearer validation to `ws_handler` mirroring `sse_events` (query-token or Authorization header), and scope the forwarded events to the caller's authenticated memory/tenant. Evidence

Needs graph at HEAD: `publish-dev` (line 519), `publish-release` (603), `publish-github-dev` (766), `publish-github` (842) all declare `needs: [test-container, integration-test]`; the chain resolves test-container→build-image→build→[lint, build-testdb]. Evidence

Migration migrations/20260524010000_event_outbox.sql creates a plain table with a partial index `idx_event_outbox_unpublished ... No DELETE/prune of published or aged rows exists outside tests.

Impact

The transactional-outbox pattern is only complete with a message relay that reads unpublished rows, dispatches them, and marks them published. Here the relay half is absent: every row stays `published_at IS NULL` forever, so (a) events written to the outbox are never delivered to any consumer, and (b) the hot table grows monotonically and unboundedly (transcript_partial alone is ~100 rows/call), degrading the `idx_event_outbox_unpublished` index and eventually the database. Two failures — silent non-delivery and table/index bloat leading to a production outage.

Fix

Implement (or wire up) an outbox relay task that periodically calls list_unpublished, publishes to the EventBus, and calls mark_published; add a retention job (or time-based partitioning) that deletes published rows past a TTL. Until a relay exists, stop writing high-volume partials to the outbox.

Schema & Migrations

6 findings

The 8 new migrations are individually well-formed — unique timestamp prefixes across all 114 files (no duplicates), no CREATE INDEX CONCURRENTLY, no COMMENT ON EXTENSION, sensible CHECK constraints and partial indexes, and good inline documentation. The defects are at the integration seams. Most severe: commit 10d2601f edited the already-applied 20260215000000_native_uuidv7.sql in place (adding a PG18 guard DO-block), which sqlx::migrate! checksum validation converts into a boot-refusing 'previously applied but has been modified' error on every deployment that migrated before mid-May — a fleet-wide upgrade brick with no repair path in the codebase. Second: all 8 new tables self-document as public/shared, but none were added to the SHARED_TABLES deny list in archives.rs, so the deny-list cloner and access-time auto-migration replicate them into every memory schema; combined with SET LOCAL search_path resolution, the ingest-stream handler's transactional outbox insert lands in the archive's private event_outbox clone for non-default memories and is silently never published. Rounding out the list: the call_sessions→archive_registry FK was dropped (to avoid lock cycles) without adding the compensating SET-NULL cleanup to archive deletion; the inference_config_audit table's attribution columns are never populated (changed_by hardcoded 'anonymous', source_ip always NULL) despite the schema's documented forensic purpose and fail-closed auth; event_outbox/inbound_dlq have no retention path and grow unboundedly; and the new override/webhook/connector tables persist raw API keys and HMAC secrets in cleartext with read-path-only redaction.

Realtime & Telephony

5 findings

The May 2026→HEAD realtime/telephony work adds a standards-shaped call stack (crates/matric-api/src/realtime/*), a Twilio Media Streams WebSocket, a Deepgram streaming ASR backend, and Twilio HMAC webhook verification. Secret hygiene is genuinely good: DeepgramConfig/CallControlEvent/MediaFrame all have hand-written Debug impls that redact payloads and keys to length/class only (the June 27 'tighten debug metadata' series is thorough and unit-tested at mod.rs:398-547; no config endpoint echoes the Deepgram key), and the Twilio incoming-webhook path enforces HMAC-SHA1 with constant-time verify. The mock ASR backend is opt-in only (REALTIME_ASR_BACKEND defaults to deepgram), not reachable by default. The material problems are in authN/authZ and resource control on the streaming surface: the /api/v1/ws WebSocket is classified RealtimeTransport, which is_public_without_bearer treats as fully public, and its handler performs no in-handler auth at all — leaking the entire cross-tenant ServerEvent bus (note titles, tags, filenames, archive names) to anonymous clients even with REQUIRE_AUTH=true (CRITICAL, violates ADR-094). The SSE /api/v1/events sibling authenticates the token but defaults to an all-memories firehose with no scope/tenant gate (HIGH). The Twilio media WebSocket has no concurrency cap around external Deepgram session creation (HIGH resource/cost exhaustion) and authorizes purely on CallSid + a 30s window with no cryptographic frame authentication (MEDIUM transcript-injection). Finally, Twilio signature URL reconstruction from proxy headers/OriginalUri is fragile under path rewriting (LOW, fail-closed availability). Findings ranked most-severe first with HEAD file:line evidence.

API Auth Surface

6 findings

The range built a genuinely well-structured centralized authorization layer (route_policy.rs inventory + auth_middleware + authorize_middleware) and the "tighten debug metadata" work is solid — the central ApiError→problem_response path scrubs Database/Internal errors to generic text with details only in logs (CWE-209 well handled), the RFC 9457 problem+json contract is applied consistently, provider API keys (Deepgram) are masked in Debug/logs, and CORS is a proper origin allowlist (no Any). The systemic weakness is that the new fail-closed authorization engine is bypassed for every route classified Public/RealtimeTransport/SystemHealth/PublicWithInlineProof, and those routes then do their own (or no) inline auth. The most serious result is /api/v1/ws, which serves the entire event-bus payload stream with no authentication at all — a direct ADR-094 fail-closed violation. SSE authenticates a token but never checks scope/tenant, so ?memory= yields authenticated cross-archive event reads; the public health-diagnostics routes honor attacker-controlled X-Fortemi-Memory (archive_routing_middleware does existence-only lookup, no ownership check) to leak arbitrary-archive metadata unauthenticated. Rate limiting is a single global bucket with no per-client/per-endpoint keying, leaving expensive inference/chat/ASR routes exposed to noisy-neighbor DoS. Lower-severity items: Twilio realtime WS relies on CallSid-knowledge without signature verification, SSE takes the bearer token in the query string, and the public compatibility endpoint discloses auth/deployment posture. Route-inventory completeness was verified — every registered route has a policy entry, so there is no fail-open un-inventoried-route gap, and forbid-insecure-auth-defaults.sh correctly guards REQUIRE_AUTH default drift and `auth.is_some()` optional-auth branching.

Webhooks & Outbox

7 findings

The webhook-receiver and event-outbox subsystem added in this range (incoming_webhook_receiver #819/#821, event_outbox #592/#844, inbound_source/inbound_dlq #833/#835) has a well-built HMAC verification/redaction surface and correct admin gating on management routes, but the durable-messaging half is materially incomplete. The most serious defect: event_outbox is enqueue-only — no production code drains it (list_unpublished/mark_published are unused outside tests) and there is no retention/partitioning, so high-volume transcript/note events are never delivered and the hot table grows unbounded. Additional gaps: the incoming-webhook HMAC has no timestamp/replay window (dedupe is opt-in and sender-controlled, CWE-294); receive_incoming_webhook does a non-atomic dual write (side effect commits, outbox emit is best-effort) unlike the correct transcript path; raw caller PII and provider credentials are persisted verbatim in event_outbox and inbound_dlq with no retention despite the codebase redacting everywhere else; the DLQ has no inspection/replay/retention path; inbound connector URLs are only scheme-checked (SSRF to metadata/internal, CWE-918, admin-gated); and connector credentials are stored plaintext at rest (CWE-312). No migration-hygiene or auth-fail-open violations were found in this dimension.

Jobs & Inference

8 findings

The jobs-worker changes in the range are almost entirely a telemetry-redaction sweep (no retry/claim semantics changed; GPU tier gating and the chat semaphore are sound fail-fast designs), but the inference runtime-config system added in May (#654/#655/#656) has real defects. The most serious is an ordering bug in POST /api/v1/inference/config: the Ollama generation-backend hot-swap runs during the merge step, before the dry_run short-circuit, the atomic-probe abort, the archive-scope branch, and DB persistence — so dry-runs mutate production, aborted atomic swaps leave torn state, and per-archive requests rewrite the global backend.
