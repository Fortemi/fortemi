## Design Decisions

### Dual-Track Architecture
- **Original Track**: User-authored content with YAML frontmatter snapshots
- **Revision Track**: AI-enhanced content (existing note_revision table)
- Both tracks maintain independent version histories

### YAML Frontmatter in Snapshots
- Tags embedded in version snapshot content as `snapshot_tags: [...]`
- Timestamp recorded as `snapshot_at: "..."`
- Self-contained for restore without external lookups

### Version Restore Behavior
- Restore creates NEW version (preserves audit trail)
- Option to also restore tags from snapshot
- Strips frontmatter before updating note_original

### Trigger-Based Versioning
- PostgreSQL trigger on `note_original` UPDATE
- Only fires when content actually changes
- Automatic cleanup of old versions beyond max_history limit
